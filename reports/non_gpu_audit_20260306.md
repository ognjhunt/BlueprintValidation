# Non-GPU Audit - 2026-03-06

## Scope

This audit focused on improvements that do not require a full GPU run:

- CPU-only test and lint health
- preflight and audit-script behavior
- config, reporting, and orchestration contracts
- maintainability risks in large Python modules

## Findings

### 1. Full CPU audit is red because the stats helper and its test disagree

`src/blueprint_validation/evaluation/stats_utils.py` returns `0.0` immediately for any constant non-zero paired delta. `tests/test_stats_utils.py` expects `None` when SciPy returns `NaN` for that case. The optimization short-circuits before the SciPy branch can run, so the full non-GPU suite currently fails.

Impact:

- `uv run pytest tests/test_stats_utils.py -q` fails.
- `pre_gpu_audit.sh` full scope fails because it runs the full non-GPU suite.
- `tests/test_pre_gpu_audit_script.py` also fails in its `full` parametrization because the nested audit inherits that failure.

Recommended action:

- Pick one contract for constant-offset paired samples and make helper + test agree.
- If `0.0` is the intended closed-form result, update the NaN-normalization test so it does not use a constant-offset input.

### 2. Audit-mode preflight still enforces provisioned runtime repos

`src/blueprint_validation/preflight.py` correctly relaxes some checks in audit mode, but it still unconditionally requires local Cosmos, DreamDojo, and OpenVLA repo paths under `/opt`. On the CPU fixture, `preflight --profile audit` failed 12 checks, mostly because those repos are not provisioned.

Impact:

- The advertised "pre-GPU" audit path is not actually lightweight.
- A CPU-only workstation can pass geometry and Stage-1 audit gates but still fail audit preflight for missing runtime repos.
- The fixture generated by `scripts/prepare_cpu_audit_fixture.py` does not override enough defaults to avoid those failures.

Recommended action:

- Move repo-path and wrapper-contract checks behind the same audit/runtime profiling used for weights and auth.
- Alternatively, make the CPU fixture explicitly disable action-boost and policy-training requirements.
- Add one real integration test that runs `blueprint-validate preflight --profile audit` against the generated CPU fixture and asserts the expected pass/fail set.

### 3. Report collection omits `s3d_wm_refresh_loop`

`src/blueprint_validation/reporting/report_builder.py` collects many stage result files, but it does not include `s3d_wm_refresh_loop`. The README describes Stage 3d as default-on in WM-only runs, so reports currently drop that artifact even when the pipeline executes it.

Impact:

- Reports can understate what actually ran.
- WM-only runs lose provenance for the refresh loop in both JSON and Markdown outputs.
- `tests/test_report_builder.py` does not cover this stage, so the omission is currently invisible.

Recommended action:

- Include `s3d_wm_refresh_loop` in `_collect_results`.
- Add a report-builder test that writes `s3d_wm_refresh_loop_result.json` and asserts it appears in the collected output.

### 4. Config typos are silently accepted

`src/blueprint_validation/config.py` builds `ValidationConfig` through a large sequence of `.get(...)` calls and does not reject unknown keys. A quick reproduction with `render.num_framez` loaded successfully and silently fell back to the default `num_frames=49`.

Impact:

- YAML typos can change experiments without any hard failure.
- This is especially risky for long-running or expensive runs where the config is the experiment definition.

Recommended action:

- Add strict unknown-key validation at the top level and within each section.
- If full strictness is too disruptive, start with warnings in audit mode and fail in runtime-cloud mode.

### 5. Stage failures lose traceback context

`src/blueprint_validation/stages/base.py` catches all exceptions and returns `StageResult(detail=str(e))`. That preserves the message but discards traceback, exception type, and local context.

Impact:

- CPU-side debugging is slower than it needs to be.
- Failures in the large stage files are harder to localize from saved artifacts alone.

Recommended action:

- Store exception type and traceback text in `StageResult.outputs` or a sidecar error artifact.
- Prefer `logger.exception(...)` over `logger.error(...)` in the generic exception path.

## Maintainability Risks

The biggest non-GPU maintainability risk is concentration of logic in a few oversized modules:

- `src/blueprint_validation/stages/s1_render.py`: `RenderStage` spans about 2778 lines
- `src/blueprint_validation/stages/s2_enrich.py`: `EnrichStage` spans about 992 lines
- `src/blueprint_validation/stages/s4_policy_eval.py`: `PolicyEvalStage` spans about 1009 lines
- `src/blueprint_validation/preflight.py`: `run_preflight` spans about 698 lines
- `src/blueprint_validation/config.py`: `load_config` spans about 1292 lines

These are not immediate bugs, but they are where CPU-only refactoring time will pay back fastest.

Recommended action:

- Split `load_config` into section parsers with explicit schemas.
- Split `run_preflight` into profile-aware check groups.
- Extract Stage-1 geometry, quality, active-perception, and audit helpers into smaller modules.

## Process Gaps

- `scripts/pre_gpu_audit.sh` does not run `geometry-canary` or `post-s1-audit`; CI adds those separately.
- The test marker taxonomy is mostly unused. `gpu` is used once; `slow` and `integration` are effectively unused.
- `mypy` is installed in `pyproject.toml` but is not configured or run anywhere.

Recommended action:

- Pull the CPU CLI audit commands into `scripts/pre_gpu_audit.sh` so local and CI gates match.
- Either start using `slow` / `integration` markers or remove them.
- Add a minimal `mypy` config and enforce it on a small high-value subset first.
